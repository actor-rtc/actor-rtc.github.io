# WebSocket 数据传输通道

## 1. 设计目标与部署模式

WebSocket 通道是 actr-runtime wire 层提供的第二种点对点传输方式，与 WebRTC DataChannel 并列。actr node 同时支持两种方向：

| 方向 | 配置项 | 典型场景 |
|------|--------|---------|
| **出站（直连发起方）** | 由信令服务器发现，无需本地配置 URL | node 在服务发现后，使用对端注册的 ws:// 地址主动建连 |
| **入站（直连接受方）** | `listen_port` | node 自身监听端口，对端 node 直接发起 WebSocket 连接 |

两种方向可同时开启。对一个 node 而言，它既可以作为出站发起方（通过发现对端的 `ws_address` 主动连接），也可以作为入站接受方（配置 `listen_port` 后由对端来连）。

### 与 WebRTC 的关系

两种通道并不互斥。runtime 在出站方向会**同时**发起 WebSocket 和 WebRTC 连接（饱和式连接策略），两者都保持 Ready 后均可用于传输；发送路径按 `data_lane_types()` 定义的候选顺序优先选用 WebRTC。

| 属性 | WebRTC DataChannel | WebSocket |
|------|--------------------|-----------|
| 路由优先级 | 高（候选列表首位） | 低（候选列表次位） |
| 建连时间 | 慢（ICE 协商，1~3s） | 快（TCP+WS 握手，<200ms） |
| 网络兼容性 | 受 UDP/防火墙限制 | 仅依赖 TCP |
| 媒体流（RTP/MediaTrack） | ✅ | ❌ |
| 其他 PayloadType | ✅ | ✅ |

## 2. 配置方式

### 2.1 `Actr.toml` 配置项

WebSocket 入站监听配置在 `[system.websocket]` 块中：

```toml
[system.websocket]
# 入站：监听来自对端节点的直连请求
listen_port = 9001
```

- `listen_port`：可选。配置后本节点以服务端模式接受入站 WebSocket 直连。
- 未配置时，入站 WebSocket 不启动；但出站 WebSocket 仍可在服务发现成功后自动建立。

> **出站 WebSocket 的 URL 来源**
>
> 出站连接的目标地址**不在** `Actr.toml` 中配置，而是来自信令服务器的服务发现响应。
> 对端 actor 在向信令服务器注册时，通过 `RegisterRequest.ws_address` 字段上报自身的 ws:// 地址。
> 发起方在 `RouteCandidatesResponse` 的 `ws_address_map` 中收到该地址后即可建连，无需任何本地模板配置。

> **典型双节点配置示例**
>
> NodeA（出站发起方，无需在 `Actr.toml` 中配置 ws 地址）：  
> NodeB 的 `Actr.toml`：`listen_port = 9001`，并在 `RegisterRequest.ws_address = "ws://nodeB.internal:9001"` 中注册直连地址。  
> NodeA 发现 NodeB 后，从 `ws_address_map` 获得地址，自动向 `ws://nodeB.internal:9001` 建连。

### 2.2 `ws_address` 的传递链

ws_address 贯穿注册、路由发现、建连三个阶段：

```
NodeB 启动时
   RegisterRequest.ws_address = "ws://nodeB.internal:9001"
        │
        ▼  actrix signaling server
   ServiceInfo.ws_address（存储于 ServiceRegistry）
        │
        ▼  RouteCandidatesOk
   ws_address_map: [WsAddressEntry { candidate_id, ws_address }]
        │
        ▼  actr runtime (send_discovery_request)
   ActrNode.discovered_ws_addresses: HashMap<ActrId, String>
        │  （Arc<RwLock<>> 与 DefaultWireBuilder 共享）
        ▼  DefaultWireBuilder::create_connections
   WebSocketConnection::new(url).with_local_id(local_id_hex)
        │
        ▼  WirePool::add_connection
   connect().await  →  WS 握手成功，ready
```

### 2.3 入站配置读取链

`listen_port` 从 TOML 逐层解析到 `ActrNode`：

```
Actr.toml [system.websocket]
   listen_port
        │
        ▼  actr-config/src/raw.rs
   RawWebSocketConfig { listen_port }
        │
        ▼  actr-config/src/parser/v1.rs  →  merge_system_config()
   Config { websocket_listen_port }
        │
        └─ websocket_listen_port  ──► actr_node.rs 步骤 1.7.6 启动 WebSocketServer
```

## 3. 出站连接（作为直连发起方）

### 3.1 URL 来源：服务发现驱动

出站 WebSocket 的 URL 由服务发现流程提供，而非本地配置的模板：

1. `ActrNode::discover_route_candidates` 调用 `send_discovery_request`，向信令服务器发送 `RouteCandidatesRequest`
2. 信令服务器在 `RouteCandidatesOk.ws_address_map` 中返回每个候选 actor 的 `WsAddressEntry { candidate_id, ws_address }`
3. runtime 从 `ws_address_map` 中提取地址（`CandidateCompatibilityInfo` 不再承载 ws_address），写入共享的 `discovered_ws_addresses` map：

```rust
// send_discovery_request 内部
let ws_addresses: HashMap<ActrId, String> = success
    .ws_address_map
    .iter()
    .filter_map(|entry| {
        entry.ws_address.as_ref()
            .map(|url| (entry.candidate_id.clone(), url.clone()))
    })
    .collect();

// discover_route_candidates 内部
let mut map = self.discovered_ws_addresses.write().await;
for (id, url) in &result.ws_addresses {
    map.insert(id.clone(), url.clone());
}
```

4. `DefaultWireBuilder` 在 `create_connections` 时查询 `discovered_ws_addresses`；若目标 actor 的地址已缓存则建立 WebSocket 连接，否则跳过（仅建 WebRTC）

### 3.2 连接建立

`DefaultWireBuilder::create_connections` 调用 `resolve_websocket_url(dest)` 从 `discovered_ws_addresses` 获取 URL，然后创建 `WebSocketConnection::new(url).with_local_id(local_id_hex)`。

`new()` 只构造对象，不建立网络连接。`with_local_id()` 附加本地节点的 ActrId（hex 编码），在握手时发送。连接对象随后通过 `WirePool::add_connection` 加入连接池，在一个独立的 tokio task 中后台异步进行以下步骤：

1. 调用 `WebSocketConnection::connect()` → 内部手动构造包含 `Host`、`Connection`、`Upgrade`、`Sec-WebSocket-Version`、`Sec-WebSocket-Key` 五个标准 WebSocket 升级头以及自定义 `X-Actr-Source-ID` 头的 HTTP 请求，通过 `tokio_tungstenite::connect_async(request)` 完成 TCP 连接和 WS 握手
2. 握手成功后，`connect()` 将 stream `split()` 成写半端（`WsSink`）和读半端
3. 调用 `spawn_dispatcher()`，启动后台帧分发任务（见第 7 节）
4. 状态置为 Ready，广播 ReadySet 更新

连接失败时，`WirePool` 按指数退避策略（默认最多 3 次，初始延迟 1s，倍数 2.0，上限 10s）自动重试。

### 3.3 `X-Actr-Source-ID` 请求头（出站侧）

`actr_node.rs` 在初始化时将本地节点的 ActrId 序列化为 protobuf 字节并 hex 编码，存入 `DefaultWireBuilderConfig.local_id_hex` 字段。`DefaultWireBuilder` 将其通过 `with_local_id()` 传入每个 `WebSocketConnection`。

`connect()` 方法使用 `tokio_tungstenite::tungstenite::http::Request` builder 手动构造完整的升级请求。**注意**：tungstenite 0.28 当收到预构建的 `http::Request` 时不会自动补全任何 WebSocket 标准头，以下 5 个头必须全部由调用方显式写入，缺少任意一个均会导致握手失败：

```
GET ws://nodeB.internal:9001 HTTP/1.1
Host: nodeB.internal:9001
Connection: Upgrade
Upgrade: websocket
Sec-WebSocket-Version: 13
Sec-WebSocket-Key: <随机生成的 base64 key，由 generate_key() 生成>
X-Actr-Source-ID: <hex bytes>
```

服务端收到连接时即可从 HTTP 头中识别对端身份，无需额外握手帧。

## 4. 服务端（入站连接）

### 4.1 服务端启动流程

当 `listen_port` 已配置，`ActrNode::start()` 在步骤 1.7.6 启动入站服务端：

1. 调用 `WebSocketServer::bind(port)`，返回 `(WebSocketServer, conn_rx)`
   - `bind` 内部执行 `TcpListener::bind(addr)` 绑定端口
   - `conn_rx` 是一个 channel 接收端，存储 `(WebSocketConnection, Vec<u8>)` 元组（连接 + 对端 ActrId 序列化字节）
2. 调用 `WebSocketGate::new(conn_rx, pending_requests, data_stream_registry)` 构建入站路由网关
3. 调用 `server.start(shutdown_token)` — 非阻塞，accept loop 在后台 task 中运行
4. 将 `WebSocketGate` 保存到 `actr_node.websocket_gate` 字段（`Option<Arc<WebSocketGate>>`）

在步骤 3.5，`WebSocketGate::start_receive_loop(mailbox)` 被调用，启动连接调度循环（每来一条新连接，触发 `spawn_connection_tasks`）。

### 4.2 对端身份识别：`X-Actr-Source-ID` 请求头

入站 WebSocket 握手是标准 HTTP Upgrade 请求。连接发起方**必须**在升级请求中携带自定义头：

```
X-Actr-Source-ID: <hex-encoded ActrId protobuf bytes>
```

服务端在 `WebSocketServer::start()` 的 accept loop 中，使用 `tokio_tungstenite::accept_hdr_async` 接受连接，并在握手回调中读取该头，进行 hex 解码得到对端的 ActrId 字节，与 `WebSocketConnection` 一起发往 `WebSocketGate`。

`accept_hdr_async` 调用时传入 `MaybeTlsStream::Plain(tcp_stream)`，生成的连接类型为 `WebSocketStream<MaybeTlsStream<TcpStream>>`，与出站模式相同——两端统一使用 `WebSocketConnection` 结构，无需区分角色。

### 4.3 入站连接的构建：`from_server_stream`

服务端拿到握手完成的 `WebSocketStream<MaybeTlsStream<TcpStream>>` 后，调用 `WebSocketConnection::from_server_stream(ws_stream)` 构建连接对象：

- 直接对 stream 调用 `split()` 得到写半端和读半端
- 调用 `spawn_dispatcher()` 启动帧分发任务

此路径不经过 `connect()` 方法，连接的 TCP+WS 握手已在 `accept_hdr_async` 中完成。

## 5. 连接管理

### 5.1 出站方向：`WirePool`

出站方向的所有连接（WebSocket 和 WebRTC）统一由 `WirePool` 管理。内部使用固定大小数组（`[Option<WireStatus>; 2]`，WebSocket=0，WebRTC=1）而非 HashMap：

- `add_connection(WireHandle)` 将连接句柄加入池，并在后台发起连接
- 连接状态通过 `watch::Sender<ReadySet>` 广播，发送方监听 `watch::Receiver` 等待可用连接
- 两种连接均保持 Ready 后都可用；发送路径根据 `data_lane_types()` 候选顺序决定优先级（WebRTC 优先）

### 5.2 入站方向：`WebSocketGate`

入站连接由 `WebSocketGate` 管理，其 `conn_rx` 字段类型为 `Mutex<Option<Receiver<(WebSocketConnection, Vec<u8>)>>>`，采用"一次性取出"模式：

- `start_receive_loop(mailbox)` 通过 `conn_rx.lock().take()` 取出 `Receiver`（此后 `Receiver` 归后台 task 所有，不可重复调用）
- 后台 task 每收到一条新连接，调用 `spawn_connection_tasks(conn, source_id, mailbox, data_stream_registry)`
- 每条入站连接生成 4 个读取 task，分别对应 4 种 `PayloadType`（`RpcReliable`、`RpcSignal`、`StreamReliable`、`StreamLatencyFirst`）

## 6. 消息发送路径

发送时，`DestTransport::send` 查询 `PayloadTypeExt::data_lane_types()` 得到该 `PayloadType` 的有序候选通道列表：

- `RpcReliable`、`RpcSignal`、`StreamReliable`、`StreamLatencyFirst`：候选为 `[WebRtcDataChannel, WebSocket]`（WebRTC 优先）
- `MediaRtp`：候选为空（`&[]`），WebSocket **不参与**媒体流路由

选中 WebSocket 通道后：

1. 调用 `WebSocketConnection::get_lane(payload_type)` 取得（或懒创建）对应 `DataLane`
2. `DataLane` 持有 `WsSink` —— 类型为 `Arc<Mutex<Option<SplitSink<WebSocketStream<MaybeTlsStream<TcpStream>>, WsMessage>>>>`
3. 将数据序列化为二进制帧（见第 8 节），调用 `sink.send(WsMessage::Binary(buf))`

---

## 7. 消息接收与路由

### 7.1 帧分发：`spawn_dispatcher`

无论是出站连接还是服务端入站连接，`WebSocketConnection` 构建完成后都会调用 `spawn_dispatcher()`，启动一个后台 task：

- 从 WebSocket 读半端持续读取二进制帧
- 解析帧头（1 字节 `payload_type` + 4 字节 BE 长度），按 `PayloadType` 路由到对应 mpsc channel（路由表为固定大小数组 `[Option<mpsc::Sender>; 5]`）
- 每种 `PayloadType` 有独立的 mpsc channel；`DataLane::recv()` 从对应 channel 中读取

### 7.2 入站路由：`WebSocketGate::handle_envelope`

对于服务端入站连接，`spawn_connection_tasks` 为每条连接的每种 `PayloadType` 启动一个读取 task，循环调用 `DataLane::recv()` 并将消息交给 `handle_envelope`：

- **RPC 类型**（`RpcReliable`、`RpcSignal`）：解码为 `RpcEnvelope`；若 `request_id` 在 `pending_requests` 中存在则唤醒等待方（Response），否则投递到 `Mailbox`（Request），`RpcSignal` 使用 `MessagePriority::High`
- **DataStream 类型**（`StreamReliable`、`StreamLatencyFirst`）：将数据投递给 `DataStreamRegistry` 进行流处理

---

## 8. 物理帧格式

每一帧为二进制 WebSocket 消息，格式固定：

```
┌─────────────────┬──────────────────────┬──────────────┐
│  payload_type   │      data_len        │     data     │
│   (1 byte)      │    (4 bytes, BE)     │   (N bytes)  │
└─────────────────┴──────────────────────┴──────────────┘
```

- `payload_type`：`PayloadType` 枚举值（0=RpcReliable, 1=RpcSignal, 2=StreamReliable, 3=StreamLatencyFirst, 4=MediaRtp）
- 同一条 TCP 连接上**复用**多个逻辑通道，由帧头字节区分

## 9. 数据流全景

```
                       ┌─────────────────────────────────┐
                       │    actrix 信令服务器              │
                       │                                  │
  NodeB 注册时 ──────► │ ServiceInfo.ws_address           │
    RegisterRequest     │   = "ws://nodeB:9001"            │
    .ws_address         │                                  │
                        │ RouteCandidatesOk                │
  NodeA 发现时 ◄─────── │   .ws_address_map               │
                        │   = [{nodeB_id, ws://nodeB:9001}]│
                        └─────────────────────────────────┘

NodeA（出站发起方）                      NodeB（入站接受方）
───────────────────────────────────      ────────────────────────────────────
ActrNode::start()                        Actr.toml: listen_port = 9001
DefaultWireBuilderConfig                 WebSocketServer::bind(9001)
  local_id_hex = hex(nodeA_id)           accept_hdr_async (后台 accept loop)
  discovered_ws_addresses (共享 Arc)      提取 X-Actr-Source-ID → hex::decode
                                         from_server_stream(ws_stream)
discover_route_candidates()              conn_tx.send((conn, nodeA_id_bytes))
  ws_address_map → discovered_ws_addresses        │
                                                  ▼
DefaultWireBuilder::create_connections   WebSocketGate::start_receive_loop(mailbox)
  resolve_websocket_url → "ws://nodeB:9001"  spawn_connection_tasks(conn, nodeA_id)
  WebSocketConnection::new(url)           4 个 PayloadType 读取 task
    .with_local_id(local_id_hex)          handle_envelope()
WirePool::add_connection                  ├─ RPC    → Mailbox（via PendingRequests）
  │                                       └─ Stream → DataStreamRegistry
connect().await
  HTTP GET ws://nodeB:9001 HTTP/1.1
  X-Actr-Source-ID: <hex>
  ────────────────────────────────►
split() + spawn_dispatcher()
Ready（广播 ReadySet）
  │
send(RpcReliable, data)
get_lane(RpcReliable) → sink.send([0x00][len][data])
```

## 10. 当前限制

| 限制 | 说明 |
|------|------|
| 不支持 `MediaRtp` | 媒体流（RTP）仅走 WebRTC MediaTrack，WebSocket 不在路由候选中（`data_lane_types()` 返回空切片） |
| 出站 URL 依赖服务发现 | 若信令服务器未返回 `ws_address`（对端未注册直连地址），出站 WebSocket 连接不会建立 |
| 入站侧无传输层加密 | `listen_port` 目前仅接受明文 TCP；TLS 支持需额外配置 |
| 需要 IP/端口直连 | 两端之间必须有可达的 TCP 路径；NAT 穿透场景需借助 WebRTC |
